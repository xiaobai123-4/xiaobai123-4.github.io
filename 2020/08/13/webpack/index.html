前端模块化

  1. 作用域
    ①全局作用域（window）
    ②局部作用域（函数内部定义的变量）
  2. 命名空间
    引入外部js时，防止命名冲突，变量覆盖，用对象包裹变量，但是引用对象方法时，对象中的方法中的值可以用”对象.属性名”随意修改
    解决办法：闭包，变量不可被外部随意修改
  3. 模块化
    代码如下： 
    (function(window){
           var name="jack";
           var age="23";
           function tell(){
               console.log("我的名字是"+name);
               console.log("我的年龄是"+age);
           }
           window.susanModule={tell};
       })(window)
       //调用方法：suanModule.tell
  4. 模块化的优点
    ①作用域封装
    ②重用性
    ③解除耦合（关系紧密度）

演化

  1. COMMONJS
    模块标准：module.exports+require()，支持该标准的有Node.js

    //通过require函数来引入
    const math=require('/math');
    //通过exports将其导出
    module.exports.getSum=function(a,b){
        return a+b;
    }

  2. ES6 MODULE
    模块标准：export+import

    //import导入
    import math from './math';
    //export导出
    export function sum(a,b){
        return a+b;
    }

  3. AMD和CMD（用的较少）
    ①AMD
    异步加载模块定义：define()

    //求和模块
    //define(名字,依赖,函数/对象)
    define('getSum',['math'],function(math){
    	return function(a,b){
    		console.log('sum:'+math.sum(a,b));
    	}
    });

    ②CMD
    同步加载模块标准：define()+require()
    a.AMD推崇依赖前置，在定义模块的时候就要声明其依赖的模块
    b.CMD推崇就近依赖，只有在用到某个模块的时候再去require

CommonJS规范 ES6
导出内容 //x.js
module.exports={
age:10,
print:function(){}
}
}
 //①x.js普通导出
export var age=10;
export function print(){};
//②x.js默认导出
export default{
var age=10;
print:function(){}
}
导入内容 //y.js
const obj=require(‘./x’)		 //①import {age,print} from (“./x”)

//②y.js
import obj from ‘./x’
obj.age
obj.print()
导出数量 只能导出一个 ①可以导出多个变量和方法
②只能导出一次
使用限制 可以使最顶层或内部代码 只能最顶层
支持情况 服务器端技术：可以被Node.js解释器直接理解 客户端技术：目前还不能被各大浏览器支持，只能使用编译器进行预处理转换为标准JS代码

ES6中的模块必须使用一个编译器(webpack)转换成ES5中代码，才能间接在浏览器中运行使用

webpack

概念
webpack的核心是用于现代JavaScript应用程序的静态模块捆绑器。当webpack处理您的应用程序时，它会在内部递归构建一个依赖关系图，其中包含应用程序需要的每个模块(可能是JS/Css/图片等)然后把这些所有模块打包成一个或多个捆绑包(资源包/bundle文件)。
用途
资源打包后，再供客户端一次性加载，减少请求次数和请求数据总大小，从而提高访问效率
注意
webpack本身用于打包静态资源，通常的使用场景是：项目初步开发完成后，运行Webpack(npm扩展包),把需要处理的静态资源打包，把得到的打包文件和HTML文件一起发不到WEB服务器(node/php/java)上供客户端访问
核心概念
1.entry：入口
2.output：输出，指定最终输出的资源包文件
3.loader：加载器，让webpack可以处理非js文件，如CSS/图片
4.mode：运行模式，可以指定production(产品模式下，输出文件最小化)或者development(开发模式，输出文件不会最小化)
5.plugins:定义扩展功能性插件

使用过程

  1. 创建客户端项目，编写HTML/CSS/JS/图片等文件
  2. 创建项目的描述文件：package.json npm init
  3. 下载webpack及其依赖的模块，并添加为当前项目的“开发阶段依赖模块” npm i webpack --save-dev
  4. 需要用到的插件可查阅官网webpack
  5. 创建webpack主配置文件：webpack.config.js
    module.exports={
        // 定义打包模式
    	mode:'production',
        // 定义打包的入口文件--通常选取依赖于很多其他文件的JS做入口
    	entry:'./src/index.js',
        // 定义打包得到的输出文件及其所在目录
    	output:{
    		path:__dirname+'/dist',
    		filename:'bundle.js'
    	},
        // 定义打包非js文件所用到的加载器(loader)
    	module:{},
    	// 定义扩展功能性插件
    	plugins:[]
    }
  6. 运行webpack,根据主配置文件对静态资源进行打包 node ./node_modules/webpack/bin/webpack/bin/webpack.js或者在package.json中配置script:"build":"webpack",用npm run build方式运行

gulp和grunt的定位是自动化的构建工具，帮程序员实现自动化去完成反复重复的工作而webpack是专业的的模块化打包工具

实例：

// 默认配置文件
module.exports = {
  // 定义打包模式
  mode: 'production', //默认值，产品模式下，输出文件最小化
  // mode:'development', //开发模式，输出文件不会最小化
  // 定义打包的入口文件--通常选取依赖于很多其他文件的JS做入口
  entry: './src/js/index.js',
  // 定义打包得到的输出文件及其所在目录
  output: {
    filename: 'app.js',
    path: __dirname + '/dist',
    publicPath: ''
  },
  // 定义打包非js文件所用到的加载器(loader)
  module: {
    rules: [
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader']
      },
      {
        test: /\.(jpg|png|gif|svg|ttf)$/i,
        use:[
          {
            loader:'file-loader',
            options:{
              name:'img/[name].[ext]?[hash]',
              publicPath: 'dist/'
            }
          }
        ]
      }
    ]
  }
  // 定义扩展功能性插件
  plugins:[]
}
